# BGE-M3 嵌入模型与 BGE-Reranker-v2-m3 重排模型

## 1. BGE 家族简介

BGE（BAAI General Embedding）是北京智源人工智能研究院发布的开源嵌入模型系列。自 2023 年首次亮相以来，BGE 在 MTEB 等中文/英文/多语言榜单上长期占据领先位置，成为中文 RAG 社区最常用的嵌入模型之一。其中 **BGE-M3** 是该系列中综合能力最强的通用模型，**BGE-Reranker-v2-m3** 则是与之配套的交叉编码器重排模型。

## 2. BGE-M3 的"M3"含义

"M3"代表三重特性：

- **Multi-Functionality（多功能）**：单一模型同时输出**稠密向量（Dense）**、**稀疏向量（Lexical / Sparse）**、**多向量（ColBERT-style Multi-Vector）** 三种表征。
- **Multi-Linguality（多语言）**：原生支持 100 多种语言，中文与英文表现尤其突出，适合跨语种检索。
- **Multi-Granularity（多粒度）**：一次支持从短句到最长 **8192 个 Token** 的上下文，远超一般嵌入模型的 512 Token 限制，非常适合检索大段落。

这三种表征可以按需组合，在一次前向传播里同时产出，显著降低了工程集成的复杂度。

## 3. 稠密、稀疏与多向量表征

- **Dense 稠密向量**：默认维度 **1024**，是最常用的语义相似度表征。使用时建议 L2 归一化，然后用余弦相似度或内积打分。
- **Sparse 稀疏向量**：类似 SPLADE，输出每个词元的权重，擅长精确词匹配与专有名词。可以写入 Milvus 的 SPARSE_FLOAT_VECTOR 字段。
- **Multi-Vector（ColBERT 式）**：为每个 Token 输出一个向量，打分时采用 MaxSim 聚合。这种表征在精排阶段精度更高，但存储成本也更大。

在工程实践中，最常见的组合是"Dense 做召回 + Sparse 做关键词兜底 + Reranker 做精排"；如果资源充裕，也可以再叠加 Multi-Vector 进一步提升精度。

## 4. 维度与归一化

BGE-M3 的稠密向量输出维度为 **1024**，模型在输出前通常已经做了归一化（L2 范数约等于 1）。对应的 Milvus 集合配置示例：

- `dim = 1024`
- `metric_type = "IP"`（等价于已归一化下的余弦相似度）
- 索引：`HNSW`，`M = 16`，`efConstruction = 200`
- 查询：`ef = 128` 起步，按召回要求微调

若嵌入服务输出未归一化，应在写入前显式做一次 `x / ||x||`，避免 IP 排序被向量模长扰动。

## 5. 上下文长度与切分建议

BGE-M3 支持最长 8192 Token，但这并不意味着我们在 RAG 场景下就应当把 chunk 切到 8K 那么长。几点经验：

1. **检索召回粒度不宜过粗**：过长的 chunk 会让向量被稀释，检索时不容易精确定位。
2. **中文常用切分**：300–800 字符 + 10%–20% 重叠，既保留语义完整性，又不至于让 top-k 候选彼此高度冗余。
3. **遇到表格、代码、列表**：优先按结构边界切，避免把一个代码块拦腰切断。
4. **长上下文优势**：当原文确有长段推理（例如法律条文、学术论文）时，BGE-M3 的 8K 能力能让单个 chunk 包含更完整的上下文。

## 6. 与其他嵌入模型的对比

- **BGE-M3 vs OpenAI text-embedding-3-large**：OpenAI 模型维度 3072，MTEB 英文略占优；BGE-M3 在中文任务上常年领先，且完全开源自部署。
- **BGE-M3 vs Cohere embed-multilingual-v3**：两者都强调多语言，Cohere 闭源、需计费；BGE-M3 的本地化部署成本优势明显。
- **BGE-M3 vs M3E / text2vec**：M3E 与 text2vec 是早期中文社区常用的选项，在长文本与多语言能力上均不及 M3。
- **BGE-M3 vs Jina Embeddings v3**：Jina v3 同样是优秀的多语言通用嵌入，BGE-M3 在 SPLADE-style 稀疏输出上更成熟。

## 7. BGE-Reranker-v2-m3 概览

BGE-Reranker-v2-m3 是与 BGE-M3 配套的交叉编码器重排模型。它的输入是 `(query, passage)` 对，输出一个标量分数表示相关性。关键特性：

- **交叉注意力打分**：query 与 passage 同时进入同一个 Transformer，精度远高于双塔嵌入。
- **多语言支持**：与 M3 一致的多语言覆盖。
- **中等尺寸**：在精度与延迟之间取得平衡，单 GPU 上可支撑百级 QPS。
- **工业界广泛采用**：许多开源 RAG 方案默认搭配 BGE-Reranker 系列作为精排。

## 8. 重排的工程定位

重排的价值在于"对候选集合做精细重排"，而不是"扩大召回"。典型的两阶段流水线：

1. **召回阶段**：BGE-M3 + 向量库检索 + 可选稀疏检索 → 得到 top-50 或 top-100 候选。
2. **重排阶段**：BGE-Reranker-v2-m3 对候选打分 → 取 top-5 或 top-10。
3. **生成阶段**：拼入 Prompt，由大模型生成答案。

这种两阶段设计的核心思想：**召回看全局但不精细，精排看局部但更严格**。如果只用双塔嵌入直接截 top-5，容易漏掉一些"语义相关但表面不匹配"的段落。

## 9. 典型参数与使用要点

使用 BGE-Reranker-v2-m3 时，常见配置：

- **批大小**：可根据 GPU 显存调整，典型 `batch_size = 8` 或 `16`。
- **最大长度**：`max_length = 512` 是常用默认；若 chunk 本身较长，可能需要截断。
- **归一化分数**：原始输出通常不是概率，可以用 sigmoid 归一化到 `[0, 1]`，便于阈值决策。
- **相关性阈值**：根据业务需要设一个 `similarity_threshold`，分数低于阈值的候选直接丢弃。

## 10. 部署与调用

BGE-M3 与 BGE-Reranker-v2-m3 都可以通过以下方式部署：

- **Hugging Face Transformers**：官方给出了示例代码，可以直接 `from_pretrained` 加载。
- **Sentence-Transformers**：对 Dense 向量提供简洁的 API。
- **FlagEmbedding 官方库**：同时支持三种表征的统一接口。
- **TEI（Text Embeddings Inference）**：Hugging Face 出品的高性能推理服务，支持批处理与动态 batching。
- **Triton / TGI / vLLM**：可做企业级封装。

本项目 RAGForce 采用 HTTP 形式调用独立部署的 embedding 与 reranker 服务，这样可以把模型推理与业务逻辑解耦，便于水平扩展与灰度升级。

## 11. 中文场景的额外建议

- **专有名词与缩写**：BGE-M3 对常见中文专有名词有良好覆盖；对极小众的术语可以借助稀疏检索或 BM25 兜底。
- **繁简体**：M3 在训练中已经见过大量繁简语料，但业务侧仍建议提前做统一转换，避免检索歧义。
- **英文混排**：中文文档中嵌入英文（例如函数名、API 名）时，M3 能较好地保留语义，无须额外处理。
- **数字与日期**：纯数值查询仍然是嵌入模型的弱项，建议结合结构化字段过滤。

## 12. 评测中的角色

在本项目的评测矩阵里，BGE-M3 与 BGE-Reranker-v2-m3 承担了两个关键角色：

- **召回基准**：`dense_only` 与 `hybrid` 配置中，BGE-M3 的稠密向量构成主通道召回。
- **精排对照**：`hybrid_rerank` 与 `full` 配置中，BGE-Reranker-v2-m3 对候选做重排，评测它带来的 Recall@k / MRR@k / nDCG@k 增益。

通过这套评测矩阵，可以直观地量化"加不加重排"带来的质量差异，为后续的模型选型和参数调优提供可信依据。
