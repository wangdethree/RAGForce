# RRF（Reciprocal Rank Fusion）融合方法详解

## 1. 什么是 RRF

RRF，全称 **Reciprocal Rank Fusion**，中文直译为"倒数排名融合"，由 Cormack 等人在 2009 年提出，原论文是针对信息检索评测任务提出的一种简单而鲁棒的多系统结果融合方法。它的核心思想非常朴素：**一个候选在任一检索通道里排名越靠前，其最终分数就应当越高**；而具体的相似度/相关度分数不参与融合，只用"排名"这个更稳健的信号。

RRF 的典型应用场景是**混合检索**：把稠密检索（Dense）与稀疏检索（Sparse / BM25）的结果合并成一份最终排序。它对分数分布差异极不敏感，不需要对两路分数做归一化，这一点让它在工程上格外受欢迎。

## 2. 公式

RRF 的打分公式如下：

```
score_rrf(d) = Σ over i in channels:  1 / (k + rank_i(d))
```

其中：

- `d` 是某个候选文档（或 chunk）。
- `rank_i(d)` 是候选 `d` 在第 `i` 路检索结果中的排名（1 表示第一名）。若候选不出现在该路结果中，则该项不参与求和。
- `k` 是一个**平滑常数**，通常取 **60**。

最终把所有候选按 `score_rrf(d)` 从大到小排序，再截取 top-k 作为融合结果。

## 3. 参数 k 的意义

常数 `k` 的作用是"压低靠前排名的优势"：

- **k 越小**：排名 1 与排名 2 的分差越大，排前几名的候选权重越高。
- **k 越大**：分差更平滑，靠后排名的候选也能通过"多路都上榜"累积出较高总分。

原论文建议 `k = 60`，后续大量工程实践也验证了这个值在大部分场景下工作良好。特别极端的场景可以在 `[10, 100]` 之间微调；没有必要太在意它的精确值。

举个直观的例子：假如 `k = 60`，那么：

- 排名 1 的贡献是 `1 / 61 ≈ 0.01639`
- 排名 2 的贡献是 `1 / 62 ≈ 0.01613`
- 排名 10 的贡献是 `1 / 70 ≈ 0.01429`
- 排名 100 的贡献是 `1 / 160 ≈ 0.00625`

可以看到，排名差异带来的分数差距是"温和递减"的——这正是 RRF 稳健性的来源。

## 4. RRF 的工作示例

假设有两路检索通道 A（Dense）与 B（Sparse），各返回 top-5：

| 排名 | A（Dense） | B（Sparse） |
|------|-----------|-------------|
| 1 | doc_01 | doc_02 |
| 2 | doc_03 | doc_01 |
| 3 | doc_05 | doc_04 |
| 4 | doc_02 | doc_05 |
| 5 | doc_04 | doc_06 |

取 `k = 60`，逐文档计算：

- `doc_01`: 1/(60+1) + 1/(60+2) = 0.01639 + 0.01613 = 0.03252
- `doc_02`: 1/(60+4) + 1/(60+1) = 0.01563 + 0.01639 = 0.03202
- `doc_03`: 1/(60+2) = 0.01613
- `doc_04`: 1/(60+5) + 1/(60+3) = 0.01538 + 0.01587 = 0.03125
- `doc_05`: 1/(60+3) + 1/(60+4) = 0.01587 + 0.01563 = 0.03150
- `doc_06`: 1/(60+5) = 0.01538

融合后的排序是：`doc_01 > doc_02 > doc_05 > doc_04 > doc_03 > doc_06`。

可以看到：**两路都进了前列的 doc_01、doc_02 明显受益**，只在单路出现的 doc_03、doc_06 分数较低。这正是 RRF 鼓励"多路共识"的体现。

## 5. RRF 的优点

1. **无需归一化分数**：Dense 的分数区间可能是 `[-1, 1]`（余弦）或 `[0, 1]`（内积归一后），Sparse 的 BM25 分数则是非归一化的正数；如果用加权求和就必须先做归一化，RRF 直接跳过这一麻烦。
2. **实现简单**：寥寥十几行代码即可实现，没有需要学习的参数。
3. **稳健**：对单路检索偶发的异常排名不敏感，多路信息互补。
4. **可扩展**：支持任意路检索通道（Dense、Sparse、结构化过滤再重打分等），写个循环就能加一路。

## 6. RRF 的局限

1. **丢失分数信息**：只用排名，如果某路分数差距悬殊（例如 top-1 是 BM25 分数 50，top-2 只有 5），RRF 不会体现这种"一骑绝尘"的情况。
2. **对不同路数不敏感**：两路召回一致时效果最佳；若一路质量明显低，RRF 会把低质路也平等计入。实际工程中可通过**只保留每路 top-K**或**按通道加权**来缓解。
3. **调参空间有限**：`k` 只有一个全局参数，不能针对某条 query 动态自适应。

## 7. 与其他融合方法的对比

- **加权打分融合（Weighted Sum）**：先把各路分数归一化到 `[0, 1]`，再按 `α·s_dense + (1-α)·s_sparse` 线性加权。精度潜力高，但需要对分数分布做细致建模，且对异常值敏感。
- **CombSUM**：直接把归一化后的分数相加，不考虑排名。
- **CombMNZ**：在 CombSUM 的基础上，额外乘以"出现路数"，鼓励跨路共识。
- **Reciprocal Rank Fusion（RRF）**：本文主角。排名驱动，稳健简洁。
- **Borda Count**：经典投票方法，把每路排名换算成分数再求和，本质与 RRF 很接近。
- **LTR（Learning to Rank）**：用机器学习把多路信号一起学一个打分函数，天花板最高但需要标注数据。

在 RAG 的混合检索场景中，**RRF 是默认选择**，理由是零配置、效果稳定、社区生态成熟；只有在指标确认 RRF 是瓶颈时，才考虑切换到 LTR 或 Cross-Encoder 精排。

## 8. 在 Milvus 中的用法

Milvus 2.4 起原生支持 Hybrid Search，可以把两路（或多路）`AnnSearchRequest` 交给服务端融合。融合算子支持两种：

- `RRFRanker(k=60)`：直接使用 RRF。
- `WeightedRanker(weight1, weight2, ...)`：按权重做加权归一打分融合。

一个典型调用模式：

1. 分别构造 Dense 路径（稠密向量字段）与 Sparse 路径（稀疏向量字段）的 `AnnSearchRequest`。
2. 调用 `collection.hybrid_search([req_dense, req_sparse], ranker=RRFRanker(60), limit=20)`。
3. Milvus 服务端完成两路召回与融合，返回统一的 top-K。

这种"服务端融合"相比客户端融合的优势是：减少一次跨网段的数据往返，尤其在候选集较大时更明显。

## 9. 在本项目中的评测定位

RAGForce 的评测矩阵包含 `dense_only`、`hybrid`、`hybrid_rerank`、`full` 四个配置。其中：

- `dense_only`：仅用 Dense 路径，baseline。
- `hybrid`：在 Dense 之上加入 Sparse 路径，并用 **RRF（k=60）** 融合。
- `hybrid_rerank`：在 hybrid 融合后增加一次交叉编码器重排（BGE-Reranker-v2-m3）。
- `full`：在 hybrid_rerank 基础上再叠加查询改写（Query Rewriting）。

通过这套矩阵，我们可以量化：

- RRF 相比单路 Dense 带来了多少 Recall@k 增益？
- 重排是否让 RRF 的排名优势进一步放大？
- 查询改写是否能在 hard 难度 QA 上帮助 RRF 更好地覆盖同义表达？

## 10. 常见误区

1. **误解"k 越大越好"**：`k` 不是越大越好，它只是一个平滑常数。盲目调大会让排名差异变得过于平缓，反而不利于排序区分度。
2. **忘记截断每路 top-K**：如果直接把两路所有候选拿来融合，排在很后面的低质候选也会累加一份贡献。工程上通常只取每路 top-100 或 top-50 再融合。
3. **误用归一化分数**：RRF 只需要排名，不需要分数；混入归一化分数反而破坏了 RRF 的稳健性。
4. **忽略去重**：两路可能各自返回同一个 chunk_id，融合时务必先按唯一主键去重再累加分数。

## 11. 小结

RRF 以一个 "k + rank" 的极简公式，在 RAG 的混合检索里占据了默认融合算法的位置。它的核心价值不在算法复杂度，而在**工程稳健性**：对分数分布不敏感、对参数不敏感、对多路扩展友好。本项目在评测链路中把 RRF 作为 `hybrid` 系列配置的融合算子，并通过指标矩阵量化其收益，为后续是否升级到更复杂的 LTR 方案提供数据支撑。
