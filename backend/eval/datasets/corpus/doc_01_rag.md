# 检索增强生成（RAG）技术概览

## 1. 什么是 RAG

检索增强生成（Retrieval-Augmented Generation，简称 RAG）是一种将**外部知识检索**与**大语言模型生成**结合起来的范式。它的核心思路是：当用户提出一个问题时，系统先从一个外部知识库中检索出与问题相关的若干段落，再把这些段落作为上下文拼接进提示词（Prompt），交由大语言模型生成最终答案。

传统的闭卷式问答完全依赖模型参数中"记住"的知识，一旦训练语料存在缺失、时效性过期，或涉及企业内部的私有文档，模型就很容易编造不存在的事实，也就是俗称的"幻觉"。RAG 通过在生成前注入真实的文档片段，有效缓解了这一问题，同时也使得模型能够回答超出其预训练知识截止日期之后的事实性问题。

## 2. RAG 的动机与应用场景

RAG 最早在学术界被明确提出时，目标场景是开放域问答。随着大语言模型能力不断增强，RAG 的应用范围已经扩展到企业知识库问答、技术文档助手、法律与合规问询、医疗咨询辅助、代码搜索与解释等众多方向。

典型应用场景包括：

- **企业内部知识库**：员工手册、规章制度、产品说明书往往保存在企业内网的 Wiki 或文件服务器里，员工需要快速获取相关条款。
- **客服与工单系统**：根据历史工单、产品文档自动回答用户问题，并附带引用出处，便于人工审核。
- **技术文档问答**：开源项目的 API 文档、使用教程、常见问题汇总，借助 RAG 可以提供"代码级"的精确回答。
- **科研文献综述**：围绕某个研究主题检索论文片段并生成综述草稿，帮助研究者快速建立领域认知。
- **法律与合规**：法条、判例、合同条款需要严格引用原文，RAG 的"可引用性"使它比纯生成模型更适合这一类场景。

## 3. RAG 的经典架构

一个成熟的 RAG 系统通常由以下模块组成：

1. **文档摄取（Ingestion）**：负责从 PDF、Word、Markdown、HTML、爬虫等异构源头把文档拉取进来，做清洗、去重、切分与元数据抽取。
2. **切分（Chunking）**：把长文档切成长度适中的段落（Chunk），常见策略包括按固定长度滑窗、按语义边界、按标题层级切分。中文文档常用的块大小在 300–800 字符之间，重叠在 10%–20% 较为合适。
3. **向量化（Embedding）**：把每个 Chunk 转成稠密向量，写入向量数据库。嵌入模型的质量直接决定召回上限。
4. **索引与存储**：向量库（Milvus、Qdrant、Weaviate、pgvector 等）负责稠密检索；关系库或 Elasticsearch 负责稀疏检索与元数据过滤。
5. **检索（Retrieval）**：给定查询，从向量库召回 top-k 候选。可选步骤包括查询改写、混合检索、重排。
6. **上下文装配（Context Assembly）**：把召回的候选按顺序拼接，控制总 Token 数不超过模型上下文窗口，必要时做去重与摘要。
7. **生成（Generation）**：把拼接好的 Prompt 交给大语言模型，生成最终答案，并附上引用来源。
8. **评测与监控**：对召回率、命中率、端到端延迟、答案质量做持续监控。

## 4. 稠密检索与稀疏检索

在检索模块内部，通常会同时使用两种检索方式：

- **稠密检索（Dense Retrieval）**：基于神经嵌入模型把文本映射到统一向量空间，再用余弦相似度或内积查找最近邻。它对语义同义、改写、跨语言场景友好，但对精确关键词、数字、代号类查询偶尔会漏召回。
- **稀疏检索（Sparse Retrieval）**：基于 BM25、TF-IDF 等词频统计方法，对精确词匹配非常敏感。在专业名词、型号、缩写、错别字少的查询上表现稳定。

现代 RAG 系统倾向于**混合检索（Hybrid Retrieval）**：同时调用稠密路与稀疏路，再用 RRF（Reciprocal Rank Fusion）或加权打分把两路结果融合起来，取长补短。

## 5. 查询改写与多查询扩展

用户的原始提问往往不够"检索友好"：口语化、歧义、过短、上下文依赖。查询改写（Query Rewriting）模块会在检索前对原始查询做变换，常见手法有：

- **HyDE**（Hypothetical Document Embeddings）：先让大模型根据问题生成一段"假答案"，再用假答案的嵌入去检索。
- **多查询扩展（Multi-Query）**：让大模型围绕原问题生成若干个不同角度的子问题，合并检索结果。
- **对话上下文融合**：在多轮对话里，把最近若干轮对话摘要进查询，消除"那个它指的是什么"之类的指代。

查询改写需要权衡召回收益与额外调用成本；当原问题已经足够明确时，应保留原始 query 作为兜底，避免改写后反而偏离意图。

## 6. 重排（Rerank）的作用

向量检索的返回结果是基于全局最近邻的近似排序，排在前几名的未必是"真正相关"的段落。重排模型通常是一个交叉编码器（Cross-Encoder），把 query 与每个候选 chunk 拼到一起喂进模型，输出一个相关性分数，再按分数重新排序。

对比两种打分方式：

- **双塔嵌入**（Bi-Encoder）：query 与 chunk 独立编码，速度快，适合从百万级数据里召回 top-100 候选。
- **交叉编码器**（Cross-Encoder）：query 与 chunk 一起输入同一个 Transformer，精度高但成本高，适合对 top-50 做精排。

工程上通常采用"双塔召回 + 交叉编码器重排"的两阶段流水线：召回只看语义粗匹配，重排在候选集合内部做精细的相关度打分。

## 7. 上下文装配与引用

检索到若干 chunk 之后，如何拼接成 Prompt 同样重要。常见策略：

- **按分数排序 + 拼接**：简单直接，但靠近上下文开头的内容往往被模型更重视，所以要把最相关的 chunk 放在显眼位置。
- **按文档聚合**：同一文档的多个 chunk 合并在一起，便于引用。
- **引用标注**：在 Prompt 中给每个 chunk 打上 `[1]`、`[2]` 形式的编号，要求模型在答案里显式引用，便于前端展示"答案来源"。
- **去重与压缩**：对内容高度重复的 chunk 做去重，对明显冗余的段落做总结，节省 Token 预算。

## 8. RAG 与微调的对比

很多团队在引入大模型时会纠结：是做 RAG 还是做微调？两者并不互斥，但选择上有明确分工：

- **RAG 适合"知识密集型"任务**：事实性问答、文档检索、可引用场景。知识变化快、规模大、更新频繁时，RAG 的"改数据不改模型"优势明显。
- **微调适合"风格或能力定制"**：例如让模型遵守特定输出格式、模仿某种语气、或者学习一类难以用 Prompt 表达的技能。
- 实际生产中常见的组合是：**基础模型 + 轻量微调（指令微调或 LoRA） + RAG**。微调让模型更"听话"，RAG 让模型更"博学"。

## 9. 常见误区与工程建议

1. **盲目追求大 top-k**：召回太多候选会稀释有效信息，反而让模型被无关内容干扰；实际经验是 top-k 控制在 5–10 为宜。
2. **忽视 chunk 质量**：切分不当会把同一段话拦腰切断，导致召回残缺；引入基于标题/段落边界的切分可以显著提升可读性。
3. **只看 Recall 不看 Precision**：评测阶段需要同时盯 Recall@k、MRR@k、nDCG@k，避免只调一项指标带偏整条链路。
4. **忽视中文分词与同义词**：BM25 依赖分词质量，中文场景下需要配合 jieba、ICU 或自定义词典。
5. **缺少离线评测集**：没有固定的 QA 集就无法做 A/B 对比；任何检索策略的改动都应该先在 offline 指标上验证。

## 10. 小结

RAG 的价值不在于某个炫酷的模型，而在于把"检索 + 生成"这条链路工程化、可控化、可监控化。一个优秀的 RAG 系统会同时关注索引质量、召回稳健性、重排精度、上下文装配、生成忠实度，以及最重要的一点——**离线与在线的评测闭环**。本项目 RAGForce 的设计正是围绕这一整条链路展开，后续章节将逐一介绍 Milvus、BGE、RRF、DeepSeek 等关键组件。
